# 开发环境（Dev）部署计划

本文档基于整体企业环境方案与 Dev 专项方案，结合当前已具备的 `infra-ops` 仓库（CI 门禁 + Deploy 工作流骨架），给出**从零到 Dev 可用的完整步骤**。不含排期，仅描述「做什么、为什么、做到什么程度算完成」。

---

## 一、总体目标与边界

### 1.1 本期要达成的状态

开发环境最终应满足以下能力：

- **两台 Dev 服务器**（`121.41.58.20`、`121.40.245.68`）完成标准化：Debian 12、Docker、统一 `deploy` 用户、基础安全基线。
- **应用可部署**：通过 GitHub Actions + Ansible，从 CI 机（`47.98.161.33`）一键部署到 Dev，过程可重复、可审计。
- **数据可连**：RDS 独立库 `app_dev`、OSS 独立前缀 `dev/` 可用，与 Test/Prod 严格隔离。
- **开发可联调**：开发人员能访问 Dev API/站点进行功能验证。
- **配置可追溯**：环境变量模板、Docker Compose、Nginx 配置均纳入 Git 管理，敏感值走 Secrets/Vault。

### 1.2 本期刻意不做的事

为避免范围膨胀，以下事项**不在 Dev 首期**：

- Prod 公网入口、正式域名、HTTPS 证书体系（Dev 可用临时 hosts 或内网访问）。
- JumpServer 堡垒机全量上线（Dev 可临时 SSH 白名单，Test 前必须收口）。
- Test/Prod 的 inventory 与部署流水线（但目录结构要预留）。
- 十台 ECS 完整 WireGuard 组网（Dev 阶段只需「能部署 + 能访问」的最小网络，本期末建议完成最小 WG）。
- 监控告警全套（可选装 node_exporter，Grafana 告警二期）。

## 二、阶段 0：准备与资产台账

这是所有后续工作的基础，未完成前不要急于写 Playbook 或跑部署。

### 2.1 确认 ECS 现状

对 Dev-01、Dev-02 两台机器逐项确认：

- 操作系统为 **Debian 12 最小化安装**（非宝塔面板环境；若已装宝塔，需规划卸载或重建）。
- 能通过 root 或临时账号 SSH 登录（bootstrap 期允许，后期关闭）。
- 记录每台机器的：**公网 IP、内网 IP（VPC）、实例 ID、所在区域、安全组 ID、是否已绑定 RAM 角色**。

产出物：一份**资产台账表**，后续 inventory、安全组、RDS 白名单都依赖这张表。

### 2.2 确认 CI 机现状

对 `47.98.161.33` 确认：

- 可作为 Ansible 控制机和 GitHub Self-hosted Runner 宿主机。
- 与 Dev 两台 ECS **同区域、同 VPC**（或至少内网互通），否则部署只能走公网，不稳定也不安全。
- 已克隆 `infra-ops` 仓库，Python/Ansible 环境可用（你当前的 `infra-ci` 环境即为此准备）。

### 2.3 创建 RDS 开发库

在现有 RDS 实例上：

- 创建数据库 `app_dev`，字符集 **utf8mb4**。
- 创建专用账号 `app_dev`，权限**仅限** `app_dev.*`（禁止 `GRANT ALL ON *.*`）。
- 将 Dev-01、Dev-02 的**内网 IP**加入 RDS 白名单（公网 IP 不要加入）。
- 记录连接串：主机（内网地址）、端口、库名、用户名；密码单独保管，不进 Git。

验收：从 Dev-01 内网 `telnet` 或 `nc` 测试 RDS 端口连通。

### 2.4 配置 OSS 开发前缀

对 `yzx-file-storage`（及需要的 `taoyouxiangzu`）：

- 确认或创建路径规范：`dev/uploads/`、`dev/static/` 等。
- 在阿里云 RAM 创建 **Dev-ECS-Role**，策略仅允许对 `dev/*` 前缀的读写，禁止访问 `test/`、`prod/` 路径。
- 将 Dev-ECS-Role 绑定到 Dev-01、Dev-02 两台 ECS（通过实例 RAM 角色，**不推荐**在机器上放长期 AK/SK）。

验收：在 Dev-01 上用 RAM 角色写入 `dev/smoke-test.txt`，确认成功且无法写入其他前缀。

### 2.5 确定应用技术栈

与业务团队确认：

- 语言与运行时（Java/Node/Go/PHP 等）。
- 容器镜像构建方式（Dockerfile 位置、多阶段构建等）。
- 镜像仓库选型：阿里云 ACR，或 CI 机私有 Harbor。
- 健康检查端点（如 `/health`）。
- 是否需要 Redis、消息队列，以及连接方式。

产出物：**技术确认单**，决定后续 Compose 模板和 Ansible 变量的具体内容。

### 2.6 配置 GitHub Environment 与 Secrets

在 `infra-ops` 仓库创建 GitHub **Environment: `dev`**，配置以下 Secrets：

| Secret | 用途 |
|--------|------|
| `ANSIBLE_SSH_PRIVATE_KEY` | CI 以 `deploy@` 用户连接 Dev 的 ED25519 私钥 |
| `ANSIBLE_SSH_KNOWN_HOSTS` | Dev 主机 SSH 指纹（可选，提高安全性） |
| `DB_PASSWORD` | `app_dev` 数据库密码 |
| `ACR_USERNAME` / `ACR_PASSWORD` | 拉取私有镜像（若使用 ACR） |
| `ANSIBLE_VAULT_PASSWORD` | ansible-vault 解密密码（若使用 Vault 加密敏感变量） |

可设为 Variable（非 Secret）的项：`APP_IMAGE`（镜像名）、`APP_TAG`（默认标签）等。

权限规则：

- `infra-ops` 的 `main` 分支启用 Branch Protection，要求 PR + `ci.yml` 全部通过才可合并。
- Dev 环境**不需要**人工审批（自动部署）。
- 有 push 权限的开发人员可触发 `workflow_dispatch` 部署。

## 三、阶段 1：服务器 Bootstrap（基础系统）

目标：两台 Dev 机具备**相同基线**，可供 Ansible 重复执行（幂等）。

### 1.1 配置安全组（Bootstrap 临时策略）

在阿里云安全组为 Dev-01、Dev-02 设置**临时**规则：

| 方向 | 规则 | 说明 |
|------|------|------|
| 入站 | TCP 22 | 来源：公司固定出口 IP + CI 机公网 IP（**临时**，WG 完成后关闭公网 22） |
| 入站 | TCP 8080/3000 等 | 来源：公司 IP 或后续 WG 网段（供开发联调） |
| 出站 | 全部允许 | 拉镜像、连 RDS、OSS |
| RDS 白名单 | Dev 内网 IP | 仅内网，不加公网 |

原则：**云安全组为主**，主机 `ufw` 为辅，避免两套规则互相冲突。

### 1.2 执行 Bootstrap 配置

通过手动操作或 Ansible `bootstrap.yml`（首次可手动，验证后固化为 Playbook）完成：

| 配置项 | 标准 |
|--------|------|
| 系统更新 | `apt update && apt upgrade`，时区 `Asia/Shanghai`，NTP 同步 |
| 用户 | 创建 `deploy`（部署专用）、`jump_ops`（预留堡垒机账号，本期可不用） |
| SSH | 仅密钥登录；禁止 root 直接登录；`deploy` 用户 sudo 限 docker 相关命令 |
| Docker | 官方源安装 Docker CE + Compose 插件（非 snap、非第三方脚本） |
| 目录结构 | `/opt/app`、`/opt/app/compose`、`/var/log/app` |
| RAM 角色 | 两台 ECS 绑定 Dev-ECS-Role |
| 防火墙 | 与安全组策略对齐，不重复封锁 |

### 1.3 配置 SSH 密钥体系

- 在 CI 机生成专用 ED25519 密钥对，**私钥**存入 GitHub Secret `ANSIBLE_SSH_PRIVATE_KEY`。
- **公钥**写入 Dev-01、Dev-02 的 `deploy` 用户 `~/.ssh/authorized_keys`。
- 记录两台机器的 SSH host key fingerprint，写入 `ANSIBLE_SSH_KNOWN_HOSTS`。

### 1.4 Bootstrap 验证

逐项确认：

- 从 CI 机：`ssh deploy@121.41.58.20` 和 `ssh deploy@121.40.245.68` 均成功。
- 两台机器：`docker run hello-world` 成功。
- 从 Dev-01：内网连通 RDS 端口。
- 从 Dev-01：通过 RAM 角色向 OSS `dev/` 写入测试文件成功。

**阶段验收**：两台机器配置一致；`bootstrap.yml` 连续执行两次无报错（幂等性）。

## 四、阶段 2：补齐 infra-ops 仓库与 CI 门禁

目标：仓库具备完整的 Ansible 目录结构，CI 能自动校验，Deploy 能通过 Self-hosted Runner 连上 Dev。

### 2.1 建立 Ansible 目录结构

在 `infra-ops` 仓库按以下逻辑结构扩展（与现有 `.github/workflows` 对齐）：

```
infra-ops/
├── .github/workflows/
│   ├── ci.yml              # 已有，继续完善
│   └── deploy.yml          # 已有，需改 Runner 类型
├── ansible/
│   ├── requirements.yml    # 已有
│   ├── inventories/
│   │   └── dev/
│   │       ├── hosts.yml           # Dev-01、Dev-02 主机清单
│   │       └── group_vars/
│   │           ├── all.yml         # 公共变量（时区、env=dev）
│   │           ├── dev_app.yml     # 应用相关变量
│   │           └── secrets.yml     # 敏感项，ansible-vault 加密
│   ├── playbooks/
│   │   ├── bootstrap.yml     # 首次：用户、Docker、目录、防火墙
│   │   ├── dev-base.yml      # 基础配置（日志、可选 node_exporter）
│   │   └── dev-app.yml       # 拉镜像、Compose 启停
│   └── roles/（或 .ansible/roles/）
│       ├── common/           # 用户、SSH、fail2ban、NTP
│       ├── docker/           # Docker 安装与配置
│       ├── nginx/            # Nginx 反代
│       ├── app_deploy/       # 通用应用部署
│       └── node_exporter/    # 可选监控
└── inventory/hosts.yml      # deploy.yml 引用的入口（可 symlink 到 inventories/dev/hosts.yml）
```

关键设计原则：

- **inventory 分目录**：`inventories/dev/` 独立，Test 后续加 `inventories/test/` 即可。
- **role 不分环境**：`common`、`docker`、`app_deploy` 等 role 全环境复用，仅 `group_vars` 不同。
- **变量命名统一**：`DB_HOST`、`OSS_BUCKET`、`ENV` 等键名三台环境一致，仅值不同。

### 2.2 编写 Inventory

`inventories/dev/hosts.yml` 定义：

| 主机名 | ansible_host | 分组 |
|--------|--------------|------|
| dev-01 | 内网 IP（bootstrap 期可暂用公网 IP） | `dev_app` |
| dev-02 | 同上 | `dev_worker` |

说明：inventory 里**优先写内网 IP**；公网 IP 仅 bootstrap 过渡期使用，WireGuard 完成后改为 WG 内网地址（`10.200.1.1`、`10.200.1.2`）。

### 2.3 编写 group_vars

`all.yml` 中定义非敏感公共变量：`env_name: dev`、`timezone: Asia/Shanghai`、`oss_bucket: yzx-file-storage`、`oss_prefix: dev` 等。

`dev_app.yml` 中定义应用层变量：`app_image`、`db_host`（RDS 内网地址）、`db_name: app_dev`、`db_user: app_dev`、`app_domains` 等。

`secrets.yml` 用 ansible-vault 加密：`db_password` 等敏感值；或在 deploy 时通过 GitHub Secrets 以 extra-vars 注入。

### 2.4 编写 Playbook 与 Role

按职责拆分：

| Playbook / Role | 职责 |
|-----------------|------|
| `bootstrap.yml` | 调用 common + docker role，完成阶段 1 所有基线 |
| `dev-base.yml` | 日志目录、Docker 网络、可选 node_exporter |
| `dev-app.yml` | 在 Dev-02 部署 Redis → Dev-01/02 部署应用 → Dev-01 配置 Nginx |
| `common` role | 用户、SSH 加固、NTP、fail2ban |
| `docker` role | Docker CE 安装、daemon 配置、deploy 用户权限 |
| `nginx` role | 反代配置，指向应用容器 `127.0.0.1:8080` |
| `app_deploy` role | 生成 `.env` 模板、拉镜像、`docker compose pull && up -d`、健康检查 |

### 2.5 完善 CI 门禁（ci.yml + 本地 Make）

远程 CI 与本地检查共用 `scripts/ci/*.sh`（单一事实来源），开发者 push 前运行 `make ci`。

| 能力 | 远程 | 本地 |
|------|------|------|
| 全量静态检查 | GitHub **CI Gate** | `make ci` |
| 依赖安装 | 各 job 内 `install-deps.sh` | `make setup` |
| Inventory 深度校验 | 未纳入 CI job | `make inventory` |

检查项：yamllint、shellcheck、ansible-lint、playbook syntax-check、docker compose config（文件缺失则 skip）、gitleaks。

Python 工具版本由 `requirements-dev.in` / `requirements-dev.txt` 锁定，与 CI Python 3.12 对齐。

**操作文档**：[docs/contributing.md](../contributing.md)、[docs/ci/README.md](../ci/README.md)、根目录 [README.md](../../README.md)。

**阶段验收**：PR 合并前本地 `make ci` 通过，且远程 **CI Gate** 绿。

### 2.6 注册 Self-hosted Runner

在 CI 机 `47.98.161.33` 上：

- 从 GitHub 仓库 Settings → Actions → Runners 获取注册 token。
- 安装 GitHub Actions Runner，标签设为：`self-hosted`、`dev`、`aliyun`。
- 使用专用系统用户 `runner`（不必 root）。
- Runner 仅服务 `infra-ops` 仓库（或组织级限制）。

关键原因：GitHub 托管 Runner（`ubuntu-latest`）位于境外，**无法直接 SSH 到国内 ECS**；必须用同区域 CI 机上的 Self-hosted Runner。

### 2.7 修改 deploy.yml

将 `deploy.yml` 中的 `runs-on: ubuntu-latest` 改为 `runs-on: [self-hosted, dev, aliyun]`，确保部署任务在 CI 机上执行。

保留现有能力：`workflow_dispatch` 手动触发、环境选择（dev/test/prod）、dry-run 模式、playbook/tags/limit 参数、Vault 解密。

### 2.8 首次 Deploy 验证

手动触发 `deploy.yml`：

- environment 选 `dev`
- playbook 选 `ansible/playbooks/bootstrap.yml`（或已固化的路径）
- dry_run 先选 `true`，确认无报错后再 `false` 正式执行

**阶段验收**：`deploy.yml` 通过 Self-hosted Runner 成功跑完 playbook（即使应用仍是占位镜像）；`ci.yml` 在 PR 时全部绿灯。

## 五、阶段 3：数据层与中间件

目标：应用容器能读写数据库、Redis、OSS。

### 3.1 完善 RDS app_dev

- 确认字符集 `utf8mb4`，排序规则 `utf8mb4_unicode_ci`。
- 确认 `app_dev` 账号权限仅限 `app_dev.*`，无跨库权限。
- 规划数据库迁移工具（Flyway/Liquibase），在 deploy 流程中执行（**仅 Dev 可自动执行**）。
- 数据策略：使用合成数据或脱敏副本，**严禁**从 Prod 直接导入或直连 Prod 库。

### 3.2 在 Dev-02 部署中间件

通过 `dev-app.yml`（或独立 task）在 Dev-02 部署：

| 组件 | 要求 |
|------|------|
| Redis | Docker 容器，仅监听内网/WG 地址，设置访问密码 |
| 消息队列（若需要） | RabbitMQ 或 Redis Stream，同样仅内网监听 |

应用连接串在 Dev 的 `.env` 中指向 **Dev-02 内网地址**（非 localhost，除非应用与 Redis 同机）。

### 3.3 确认 OSS 访问

- 路径规范落地：`dev/uploads/`、`dev/static/`。
- 应用通过 ECS RAM 角色访问 OSS，不在 `.env` 中放 AK/SK。
- 可选：配置 Dev 对象 30 天生命周期自动清理（节省存储费用）。

### 3.4 数据层验证

- 应用容器（或临时测试脚本）能连接 `app_dev` 并执行读写。
- 能连接 Dev-02 Redis 并 set/get。
- 能向 OSS `dev/` 前缀上传和读取文件。

**阶段验收**：三项数据层连通性全部通过，日志无连接错误。

## 六、阶段 4：应用部署与 Nginx

目标：业务应用能在 Dev 环境运行，开发人员可访问并联调。

### 4.1 确定应用仓库与镜像发布模式

推荐**模式 1**（职责清晰）：

```
app-repo（业务代码）
    │  push develop 分支
    ▼
构建 Docker 镜像 → 推送到 Registry（ACR 或 CI 机 Harbor）
    │  repository_dispatch 或 workflow 触发
    ▼
infra-ops deploy.yml（environment=dev）
    │  Ansible 更新 IMAGE_TAG
    ▼
Dev-01/02 docker compose pull && up -d
```

`infra-ops` 负责机器与基础设施；`app-repo` 负责构建镜像。两者通过**镜像标签契约**解耦。

### 4.2 配置镜像仓库

- 若用阿里云 ACR：在 Dev ECS 或 CI 机配置登录凭证（优先 RAM 角色，次选只读 AK 放 Secrets）。
- 若用 CI 机 Harbor：在 `47.98.161.33` 部署私有 Registry，Dev ECS 通过内网拉取。
- 镜像标签规范：`app:dev-<git-sha>` 或 `app:develop-latest`（Dev 可用浮动标签，Test/Prod 必须固定 SHA）。

### 4.3 Dev-01 部署模型

| 层 | 说明 |
|----|------|
| Nginx | 容器或宿主机 Nginx，反代到应用容器 `127.0.0.1:8080` |
| 应用容器 | Docker Compose 定义，配置健康检查 `/health` |
| 环境变量 | Ansible 模板（`.env.j2`）生成 `/opt/app/compose/.env`，真值来自 Secrets |

### 4.4 Dev-02 部署模型

| 服务 | 说明 |
|------|------|
| Worker | 消费队列，连接同一 `app_dev` 数据库 |
| Redis | 阶段 3 已部署，Worker 通过内网连接 |

### 4.5 建立发布流程

完整发布链路：

1. 开发在 `app-repo` 提交到 `develop` 分支。
2. `app-repo` CI 构建镜像并推送到 Registry。
3. 触发 `infra-ops` 的 `deploy.yml`（`repository_dispatch` 或手动），传入新 `IMAGE_TAG`。
4. Ansible 更新 Dev 上的 `.env` 中 `IMAGE_TAG`。
5. 执行 `docker compose pull && docker compose up -d`。
6. 等待健康检查 `/health` 返回 200。
7. 在 GitHub Actions Summary 输出部署结果。

### 4.6 配置开发访问方式

Dev 无公网域名时，任选一种（可组合）：

| 方式 | 适用场景 |
|------|----------|
| `/etc/hosts` 指向 Dev-01 公网 IP | 最快，bootstrap 期 |
| WireGuard + 内网 DNS（`dev-app.internal`） | 推荐稳定方案（阶段 5 完成后） |
| 公司办公网与 ECS 同 VPC | 直接用内网 IP + 端口 |

### 4.7 应用部署验证

- 浏览器或 curl 访问 Dev API，返回正常响应。
- 容器日志无 DB/OSS/Redis 连接错误。
- 连续部署两次（相同或不同镜像），服务不长时间宕机。
- 异步任务：向队列投递消息，Dev-02 Worker 成功消费。

**阶段验收**：开发人员能在 10 分钟内从代码提交到 Dev 环境生效（自动或半自动）。

## 七、阶段 5：网络收口（WireGuard 最小组网）

目标：Dev 环境不再依赖公网 SSH，运维与部署走内网隧道。

可与阶段 4 并行，但建议在 Dev 应用能跑通后再做，避免网络变更影响排障。

### 5.1 部署最小 WireGuard 拓扑

```
Hub 47.97.19.58 (10.200.0.1)  ← 唯一 WG Server，公网开放 UDP 51820
    ├── CI  47.98.161.33  (10.200.0.2)
    ├── Dev-01            (10.200.1.1)
    ├── Dev-02            (10.200.1.2)
    └── 开发人员笔记本     (10.200.10.x)
```

Hub 上安装 WireGuard Server；其余节点均为 Peer，`PersistentKeepalive = 25`。

### 5.2 配置 AllowedIPs 隔离

| 主体 | 可路由网段 |
|------|------------|
| 开发人员 WG Client | `10.200.0.0/24`、`10.200.1.0/24`、`10.200.2.0/24`（不含 Prod） |
| 运维人员 WG Client | `10.200.0.0/16`（全网） |
| Dev ECS Peer | `10.200.0.0/24`、`10.200.1.0/24` |
| CI ECS Peer | `10.200.0.0/16`（需访问各环境部署） |

**禁止** Dev Peer 路由到 `10.200.3.0/24`（Prod 网段）。

### 5.3 变更清单

| 项 | 操作 |
|----|------|
| Ansible | 增加 `wireguard` role，inventory 中 `ansible_host` 改为 WG IP |
| 安全组 | Dev SSH 入站改为仅 Hub 内网 IP 或 `10.200.0.0/16` |
| GitHub Runner | deploy 通过 WG 地址（`10.200.1.1`）连 Dev |
| 开发访问 | 开发人员安装 WG Client，访问 `http://10.200.1.1` 或内网 DNS |
| 公网 SSH | **关闭** Dev-01、Dev-02 的公网 22 端口 |

### 5.4 网络收口验证

- 关闭 Dev 公网 SSH 后，从 CI 机经 WG 执行 `deploy.yml` 仍成功。
- 开发人员经 WG 能访问 Dev API。
- Dev ECS 无法 ping 通 Prod 网段（`10.200.3.0/24`）。

**阶段验收**：公网 SSH 关闭后，部署与联调均正常。

## 八、日常运维流程

### 8.1 Playbook 执行顺序

**首次上线**（按顺序执行一次）：

1. `bootstrap.yml` — 两台 Dev 基线
2. `dev-base.yml` — Docker 网络、日志目录、可选 node_exporter
3. `dev-app.yml` — Dev-02 Redis → 应用 → Dev-01 Nginx

**日常发布**（每次代码更新）：

- 仅执行 `dev-app.yml`（或带 `--tags` / `--limit` 限定主机）

**变更基础配置**（PR 合并后）：

- 执行 `dev-base.yml`

### 8.2 环境变量规范

| 规则 | 说明 |
|------|------|
| 命名 | 全大写 `DB_HOST`，与 Test/Prod 键名一致，仅值不同 |
| 存放 | 模板在 Git（`.env.j2`）；真值在 GitHub Secrets / ansible-vault |
| 禁止 | 含密码的 `.env` 文件提交到 Git |
| 环境标记 | `ENV=development`，避免误连 Prod RDS |

Dev 特有开关（建议在变量中显式设置）：

- 日志级别 `DEBUG`
- Swagger/API 文档开启
- 邮件/短信走沙箱或 Mock

### 8.3 回滚与应急

| 场景 | 操作 |
|------|------|
| 新镜像异常 | GitHub 手动触发 deploy，传入上一稳定 `IMAGE_TAG` |
| 数据库迁移失败 | 修复 migration 脚本；Dev 可 drop 库重建（无生产数据） |
| Ansible 配错 | `git revert` + 重新 deploy |
| ECS 故障 | 用 `bootstrap.yml` 重建新机，更新 inventory 后重新部署 |

Dev 应养成「迁移可重跑、数据可重建」的习惯。

## 十、风险与注意事项

| 风险 | 应对 |
|------|------|
| inventory 写死公网 IP | Bootstrap 后立即记录，WG 完成后改为内网 IP |
| Runner 单点 | Dev 可接受；文档记录手工 deploy 备用步骤 |
| RDS 与 Dev 不在同区 | 延迟高；长期应将 Dev ECS 与 RDS 放同区域（`116.62.228.105` 已与 RDS 同区，Dev 两台需确认） |
| 应用与 infra 仓库脱节 | 固定「镜像构建 → dispatch deploy」契约，文档化 |
| 过早上 JumpServer | Dev 可延后；**Test 前必须收口** |
| 误用 Prod OSS/RDS | 台账与 `env_name` 变量双重检查；`ENV=development` 强制标记 |
| deploy.yml 仍用托管 Runner | 必须改为 Self-hosted，否则无法连国内 ECS |

## 十一、与 Test/Prod 的衔接（Dev 期就要做对）

| 做法 | 说明 |
|------|------|
| inventory 分目录 | 已有 `inventories/dev/`，Test 加 `inventories/test/` |
