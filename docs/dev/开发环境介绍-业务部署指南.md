## 三、仓库分工（必读）

| 仓库 | 职责 | 谁改 |
|------|------|------|
| **app-repo** | 业务代码、Dockerfile、单元测试、数据库迁移脚本、应用 CI（构建镜像） | 业务开发 |
| **infra-ops** | ECS 基线、Inventory、Ansible Playbook、基础设施 CI、部署编排 | infra / 运维 |

**不要**在 app-repo 里改服务器 SSH、安全组、WireGuard；**不要**在 infra-ops 里写业务逻辑。

### 目标发布流程（规划态）

```text
1. 开发在 app-repo 提交到 develop 分支
2. app-repo CI 构建 Docker 镜像并推送到镜像仓库（ACR 或私有 Registry）
3. 触发 infra-ops 的 deploy（或 app-repo 直接调部署 API）
4. Ansible 更新 Dev 上的镜像标签与环境变量
5. docker compose pull && up -d
6. 健康检查 /health 通过
```

> **当前**：步骤 1～2 由业务团队自建；步骤 3～6 的 `dev-app.yml` 等 Playbook **尚未落地**，发布前请与 infra 确认是「临时手工部署」还是「等待流水线接通」。

## 五、数据库（MySQL / RDS）

Dev 使用阿里云 **RDS MySQL 8.0** 上的**独立库**，与生产库隔离（方案 A，已验收）。

### 5.1 连接参数（应用配置用）

| 变量名（建议） | 值 | 备注 |
|----------------|-----|------|
| `DB_HOST` | `rm-bp1wjjf373l7t331v.mysql.rds.aliyuncs.com` | **必须用内网域名** |
| `DB_PORT` | `3306` | |
| `DB_NAME` | `app_dev` | |
| `DB_USER` | `app_dev` | 仅有权访问 `app_dev` 库 |
| `DB_PASSWORD` | 向 infra 申请 | 不进 Git；CI 用 GitHub Secret `DB_PASSWORD` |

**JDBC 示例**（Java）：

```text
jdbc:mysql://rm-bp1wjjf373l7t331v.mysql.rds.aliyuncs.com:3306/app_dev?useUnicode=true&characterEncoding=utf8mb4&serverTimezone=Asia/Shanghai
```

**注意**：

- 不要使用带 `no` 后缀的**外网**域名（`...vno...`）。
- 从**办公网笔记本**无法直连 RDS（除非你在 VPC 内）；应用在 **ECS 上运行**时自动走 VPC 内网。
- `app_dev` 账号**不能**访问 `yzx`、`risk` 等生产库（已验收隔离）。

### 5.2 Schema 迁移

- 迁移脚本放在 **app-repo**（Flyway / Liquibase / 框架自带迁移均可）。
- Dev 环境允许自动执行迁移；**禁止**连接或导入生产数据。
- 字符集请使用 **utf8mb4**。

详情：[RDS 实例现状与 Dev 规划](../rds/20260615-RDS-MySQL-实例现状与Dev规划.md) · [阶段 G 验收报告](../acceptance/20260615-阶段G-Dev-RDS-app_dev验收.md)

## 十、如何部署业务服务

### 10.1 部署前检查清单

- [ ] 已拿到 `app_dev` 数据库密码（或 CI Secret 已配置）
- [ ] Dockerfile 可在本地构建通过
- [ ] 健康检查路径约定（建议 `/health` 返回 200）
- [ ] 监听端口与 infra 约定一致（默认应用 **8080**，Nginx 反代 80）
- [ ] 若依赖 Redis：已确认 dev-02 状态
- [ ] 若依赖 OSS：已确认 RAM 角色已绑定
- [ ] 已与 infra 确认当前发布方式（手工 vs 流水线）

### 10.2 方式 A：等待标准流水线（推荐目标态）

1. app-repo 配置 CI：构建镜像并打标签 `app:dev-<git-sha>`  
2. 镜像推送到团队 Registry（ACR 等）  
3. 触发 infra-ops `deploy.yml`（GitHub Actions，environment=`dev`）  
4. infra 执行 `dev-app.yml` 更新 Compose 与 `.env`  

> Runner 与 `dev-app.yml` 接通前，本方式可能不可用，请向 infra 确认。

### 10.3 方式 B：开发期临时部署（需 infra 授权）

在 **已接入 WireGuard** 的前提下，由有权限同事在 **ci-01** 上：

1. SSH：`ssh deploy@10.200.0.2`（或私网 `172.21.226.38`）  
2. 将镜像拉到 dev-01，或 `docker compose` 于 `/opt/app/compose` 启动  
3. 环境变量写入 `/opt/app/compose/.env`（**勿提交 Git**）  

**禁止**：

- `scp` 二进制手工上生产（Prod 将来同样禁止）  
- 在未授权情况下直接改 Nginx / 系统服务  
- 在 Hub 上部署业务容器  

### 10.4 数据库初始化

1. 在 app-repo 提供迁移脚本  
2. 首次部署前在 Dev 执行迁移（本地连库需通过 ECS 跳板或请 infra 代执行）  
3. Dev 库可重建；**勿**把迁移直接对生产库执行

## 十一、运维边界与「三条铁律」

| 规则 | 对开发的含义 |
|------|----------------|
| **环境隔离** | 只用 `app_dev`、OSS `dev/`；不要连 prod 库 |
| **发布走 CI/CD** | 不要长期依赖手工改服务器上的 jar/容器 |
| **运维走内网通道** | SSH 经 WG 或未来堡垒机；不要要求开放 `0.0.0.0/0` SSH |

问题反馈、开白名单、申请 WG Peer / DB 密码：联系 **infra / 运维**（勿在群里发密码明文）。
