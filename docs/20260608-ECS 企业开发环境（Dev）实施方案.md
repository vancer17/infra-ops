# 开发环境（Dev）实施方案

基于前述整体架构与你已有的 **`infra-ops`（Ansible + GitHub Actions）** 仓库，本方案只聚焦 **「先把 Dev 跑通」**，并为 Test/Prod 预留扩展点。原则：**先可用、再收口、再审计**。

---

## 一、Dev 环境目标与边界

### 1.1 本期要做到什么

| 目标 | 说明 |
|------|------|
| 两台 Dev 机标准化 | Debian 12、Docker、统一 `deploy` 用户、基础安全 |
| 应用可部署 | 经 GitHub Actions + Ansible 一键部署到 Dev |
| 数据可连 | RDS `app_dev` 库、OSS `dev/` 前缀可用 |
| 开发可访问 | 开发人员能访问 Dev API/站点进行联调 |
| 配置可追溯 | 环境变量、Compose、Nginx 配置均在 Git 中 |

### 1.2 本期刻意不做（避免 scope 膨胀）

| 暂不实施 | 原因 |
|----------|------|
| Prod 入口、公网域名 | 与 Dev 无关 |
| JumpServer 全量上线 | Dev 可先直连/临时 SSH  bootstrap，堡垒机在 Test 前收口 |
| Test/Prod inventory | 后续加文件即可 |
| 完整 WG 十台组网 | Dev 阶段只需 **能部署 + 能访问** 的最小网络 |
| 监控/Grafana 全套 | 可选装 `node_exporter`，告警二期 |

## 二、总体架构（Dev 阶段）

```
开发人员
    │
    ├── 方式 A：公司 IP / 临时 SSH ──→ Dev ECS（bootstrap 期，1～2 周）
    │
    └── 方式 B：WireGuard ──→ 10.200.1.0/24（推荐本期末完成）

应用代码仓 (app-repo)
    │  push develop
    ▼
GitHub Actions (deploy.yml, environment=dev)
    │  SSH/Ansible over 内网或 WG
    ▼
┌─────────────────────────────────────┐
│  Dev-01 (121.41.58.20)              │
│  - Nginx / 应用容器                  │
│  - 连接 RDS app_dev、OSS dev/        │
├─────────────────────────────────────┤
│  Dev-02 (121.40.245.68)            │
│  - Worker / Redis / 次要服务        │
└─────────────────────────────────────┘
         │                    │
         ▼                    ▼
    RDS app_dev          OSS dev/
```

**Dev 阶段的安全立场**：允许比 Prod 宽松（如 Dev 可临时开 SSH 白名单），但 **数据库和 OSS 仍按环境隔离**，避免坏习惯延续到 Test/Prod。

## 三、`infra-ops` 仓库应补齐的内容（逻辑结构）

在不写具体代码的前提下，你的仓库建议按以下结构扩展（与现有 `.ansible/`、`workflows` 对齐）：

```
infra-ops/
├── .github/workflows/
│   ├── ci.yml              # 已有：lint、syntax-check
│   └── deploy.yml          # 扩展：dev 环境部署
├── ansible/
│   ├── requirements.yml
│   ├── inventories/
│   │   └── dev/
│   │       ├── hosts.yml           # Dev-01、Dev-02
│   │       └── group_vars/
│   │           ├── all.yml         # 公共变量（时区、DNS、env=dev）
│   │           ├── dev_app.yml     # 应用相关
│   │           └── secrets.yml       # 不进 Git，用 ansible-vault 或 CI Secrets 注入
│   ├── playbooks/
│   │   ├── bootstrap.yml     # 首次：用户、Docker、目录、防火墙基线
│   │   ├── dev-base.yml      # Dev 基础配置（每次部署前可跑）
│   │   └── dev-app.yml       # 拉镜像、Compose 启停
│   └── roles/                  # 或在 .ansible/roles
│       ├── common/             # 用户、SSH、fail2ban、NTP
│       ├── docker/
│       ├── nginx/
│       ├── app_deploy/         # 通用应用部署 role
│       └── node_exporter/    # 可选
```

**与应用仓库的关系**

| 仓库 | 职责 |
|------|------|
| `infra-ops` | 机器、Docker、Nginx、环境变量模板、部署 Playbook |
| `app-repo`（业务代码） | 构建镜像；`develop` 分支触发部署；或触发 `infra-ops` workflow |

两种常见模式（二选一，Dev 阶段定一种）：

- **模式 1**：`app-repo` 构建镜像推 Registry → 触发 `infra-ops` deploy workflow 传 `IMAGE_TAG`
- **模式 2**：`infra-ops` 的 deploy 只负责基础设施；`app-repo` 自带 workflow 调 Ansible API/SSH（Dev 建议模式 1，职责清晰）

## 四、分阶段实施（建议 2～3 周）

### 阶段 0：准备与台账（第 0～1 天）

| 序号 | 任务 | 产出 |
|------|------|------|
| 0.1 | 确认 Dev 两台 ECS：Debian 12 最小化安装，**未装宝塔** | 两台可 SSH |
| 0.2 | 记录每台 **公网 IP、内网 IP、实例 ID、安全组 ID** | 资产表 |
| 0.3 | 在阿里云创建 RDS 库 `app_dev`、账号 `app_dev` | 连接串（仅内网） |
| 0.4 | OSS 确认前缀 `dev/`，创建 RAM 角色 **Dev-ECS-Role**（仅 `dev/*` 读写） | RAM 策略文档 |
| 0.5 | 准备 GitHub **Environment: `dev`**，配置 Secrets | 见第五节 |
| 0.6 | 确定应用栈（Java/Node/Go/PHP）与镜像 Registry 地址 | 技术确认单 |

---

### 阶段 1：Bootstrap 基础系统（第 1～3 天）

**目标**：两台 Dev 机具备相同基线，可供 Ansible 重复执行。

#### 1.1 安全组（Dev 临时策略）

| 规则 | 说明 |
|------|------|
| 入站 TCP 22 | 来源：公司固定 IP + CI 机公网 IP（**临时**） |
| 入站应用端口 | 来源：公司 IP 或 WG 网段（如 8080、3000） |
| 出站 | 允许（拉镜像、连 RDS、OSS） |
| RDS 白名单 | 加入 Dev-01、Dev-02 **内网 IP** |

#### 1.2 手动或 Ansible `bootstrap.yml` 完成项

| 项 | 标准 |
|----|------|
| 系统 | `apt update`，时区 `Asia/Shanghai`，NTP 同步 |
| 用户 | 创建 `deploy`（部署用）、`jump_ops`（预留堡垒机，可先不用） |
| SSH | 仅密钥；`PermitRootLogin no`；`deploy` 可 sudo（限 docker 命令） |
| Docker | 官方源安装 Docker CE + Compose 插件 |
| 目录 | `/opt/app`、`/var/log/app`、`/opt/app/compose` |
| 防火墙 | `ufw` 或安全组为主；主机防火墙与云安全组 **二选一为主**，避免重复混乱 |
| RAM 角色 | 两台 ECS 绑定 **Dev-ECS-Role**（访问 OSS） |

#### 1.3 验证

- 从 CI 机或笔记本：`ssh deploy@121.41.58.20` 成功  
- `docker run hello-world` 成功  
- 从 Dev-01：`telnet/nc` RDS 端口通  
- 用 RAM 角色测试 OSS 写入 `dev/smoke-test.txt`  

**阶段验收**：两台机器配置一致；`bootstrap.yml` 可重复执行（幂等）。

---

### 阶段 2：Ansible Inventory 与 CI 门禁（第 2～4 天）

#### 2.1 建立 `inventories/dev`

| 主机名 | ansible_host | 分组 |
|--------|--------------|------|
| dev-01 | 内网 IP 或公网 IP（bootstrap 期） | `dev_app` |
| dev-02 | 同上 | `dev_worker` |

**建议**：inventory 里优先写 **内网 IP**；公网仅 bootstrap 用，后期改 WG IP。

#### 2.2 `group_vars` 关键变量（概念层）

| 变量类 | 示例用途 |
|--------|----------|
| `env_name: dev` | 全 playbook 标记 |
| `app_image` / `app_tag` | 容器镜像 |
| `db_host` | RDS 内网地址 |
| `db_name` / `db_user` | `app_dev` |
| `oss_bucket` / `oss_prefix` | `yzx-file-storage` / `dev` |
| `app_domains` | `dev-app.internal` 或临时 hosts |

敏感项：`db_password`、OSS 若不用 RAM 的 AK（不推荐），放 **GitHub Secrets + Ansible extra-vars**，或 **ansible-vault**。

#### 2.3 扩展 `ci.yml`

在合并 PR 时至少执行：

- `yamllint` / `ansible-lint`  
- `ansible-playbook --syntax-check` 针对 `dev-*.yml`  
- 可选：`ansible-playbook dev-base.yml --check`（dry-run，需 CI 能连 Dev 或 mock）

#### 2.4 扩展 `deploy.yml`（Dev）

| 配置项 | 建议 |
|--------|------|
| 触发 | `push` 到 `main` 且路径 `ansible/**`，或 `workflow_dispatch`；业务侧 `develop` 通过 `repository_dispatch` 触发 |
| Environment | GitHub **Environment: dev** |
| Runner | **Self-hosted** 在 47.98.161.33（能访问 Dev 内网/WG） |
| 步骤 | checkout → 装 Ansible 依赖 → 部署 SSH key → `ansible-playbook dev-app.yml` |

**为何用 Self-hosted Runner**：GitHub 托管 Runner 无法直接 SSH 到国内 ECS 内网；CI 机与 Dev 同区域、同 VPC 或经 WG 最稳。

**阶段验收**：手动触发 `deploy.yml` 能成功跑完 playbook（即使应用仍是占位镜像）。

### 阶段 3：数据层与中间件（第 4～6 天）

#### 3.1 RDS `app_dev`

| 项 | 说明 |
|----|------|
| 字符集 | `utf8mb4` |
| 权限 | `app_dev` 仅 `app_dev.*` |
| 迁移 | Flyway/Liquibase 在 deploy 流程中执行（**仅对 dev** 可自动） |
| 数据 | 合成数据或脱敏副本，**禁止**直连 Prod 数据 |

#### 3.2 Dev-02 中间件

| 组件 | 部署方式 |
|------|----------|
| Redis | Docker 容器，仅监听内网/WG，设密码 |
| 队列（若需要） | RabbitMQ / Redis Stream，同上 |

应用连接串在 Dev 的 `.env` 中指向 Dev-02 内网地址。

#### 3.3 OSS

| 项 | 说明 |
|----|------|
| 路径规范 | `dev/uploads/`、`dev/static/` |
| 访问 | 应用通过 RAM 角色；本地调试可用临时 STS |
| 生命周期 | Dev 对象 30 天自动清理（可选，省费用） |

**阶段验收**：应用容器（或临时 curl 脚本）能读写 DB、Redis、OSS。

### 阶段 4：应用部署与 Nginx（第 6～10 天）

#### 4.1 镜像与 Registry

| 项 | 说明 |
|----|------|
| Registry | 阿里云 ACR 或 CI 机私有 Harbor（47.98.161.33） |
| 标签 | `app:dev-<git-sha>` 或 `app:develop-latest` |
| Dev 拉取 | ECS 登录 Registry（ACR 用 RAM 或只读 AK 放 Secrets） |

#### 4.2 部署模型（Dev-01）

| 层 | 说明 |
|----|------|
| Nginx 容器或宿主机 Nginx | 反代到应用容器 `127.0.0.1:8080` |
| 应用容器 | Compose 定义，健康检查 `/health` |
| 环境变量 | 由 Ansible 模板生成 `/opt/app/compose/.env` |

#### 4.3 部署模型（Dev-02）

| 服务 | 说明 |
|------|------|
| Worker | 消费队列，连同一 `app_dev` |
| Redis | 已阶段 3 部署 |

#### 4.4 发布流程（目标态）

```
开发提交 app-repo develop
    → 构建镜像并推送
    → 触发 infra-ops deploy (dev)
    → Ansible 更新 .env 中 IMAGE_TAG
    → docker compose pull && up -d
    → 健康检查
```

#### 4.5 访问方式（Dev 无公网域名时）

| 方式 | 适用 |
|------|------|
| `/etc/hosts` 指向 Dev-01 公网 IP | 最快，bootstrap 期 |
| WireGuard + `dev-app.internal` | 推荐稳定方案 |
| 仅内网 IP + 端口 | 公司办公网与 ECS 同 VPC 时 |

**阶段验收**：浏览器或 curl 访问 Dev API 返回正常；日志无 DB/OSS 连接错误；重复部署不宕机或短暂可接受。

### 阶段 5：网络收口（第 10～14 天，可与阶段 4 并行）

Dev 能用后，建议 **本期末** 完成最小 WireGuard，避免 Dev 长期公网 SSH。

#### 5.1 最小 WG 拓扑（Dev 阶段）

```
Hub 47.97.19.58 (10.200.0.1)  ← WG Server
    ├── CI 47.98.161.33 (10.200.0.2)
    ├── Dev-01 (10.200.1.1)
    ├── Dev-02 (10.200.1.2)
    └── 开发人员笔记本 (10.200.10.x)
```

#### 5.2 变更项

| 项 | 操作 |
|----|------|
| Ansible | 增加 `wireguard` role；inventory 改用 WG IP |
| 安全组 | Dev SSH 改为仅 **Hub 内网 IP** 或 **10.200.0.0/16**（若云支持 WG 源） |
| GitHub Runner | deploy 走 WG 地址连 Dev |
| 开发访问 | 安装 WG Client，访问 `http://10.200.1.1` 或内网 DNS |

**阶段验收**：关闭 Dev 公网 22 后，仍可通过 WG + Ansible 部署成功。

## 六、Ansible Playbook 执行顺序（Dev 日常）

**首次上线**

1. `bootstrap.yml` — 两台 Dev  
2. `dev-base.yml` — Docker 网络、日志目录、node_exporter（可选）  
3. `dev-app.yml` — Redis on Dev-02 → 应用 on Dev-01/02 → Nginx  

**日常发布**

- 仅 `dev-app.yml`（或带 tag 的 limit 主机）  

**变更基础配置**

- `dev-base.yml`（PR 合并后手动或自动）

## 九、回滚与应急（Dev）

| 场景 | 操作 |
|------|------|
| 新镜像异常 | GitHub 手动触发 deploy，传入上一稳定 `IMAGE_TAG` |
| 数据库迁移失败 | 修复 migration 脚本；Dev 可 drop 库重建（无生产数据） |
| Ansible 配错 | `git revert` + 重新 deploy |
| ECS 故障 | 用相同 bootstrap playbook 重建新机，恢复 inventory |

Dev 应养成 **「迁移可重跑、数据可重建」** 的习惯，降低心理负担。

## 十、风险与注意点

| 风险 | 应对 |
|------|------|
| inventory 写死公网 IP | 第二期改内网/WG IP，bootstrap 后立即改 |
| Runner 单点 | Dev 可接受；记录手工 deploy 备用步骤 |
| RDS 与 Dev 不在同区 | 延迟高；长期把 Dev ECS 与 RDS 放同区域 |
| 应用与 infra 仓库脱节 | 固定「镜像构建 → dispatch deploy」契约 |
| 过早上 JumpServer | Dev 可延后；**Test 前必须收口** |
| 用 Prod OSS/RDS | 台账与变量双重检查 `env_name` |

## 十一、与后续 Test/Prod 的衔接（Dev 里就要做對）

| 做法 | 说明 |
|------|------|
| inventory 分目录 | 已有 `inventories/dev/`，Test 加 `test/` 即可 |
| role 复用 | `common`、`docker`、`app_deploy` 不分环境，仅 vars 不同 |
| workflow 用 Matrix/Environment | `deploy.yml` 增加 `environment: test|prod` |
| 变量命名统一 | `group_vars/all.yml` 结构三台一致 |
| `deploy` 用户 | 全环境同名，CI 密钥按环境分 GitHub Environment |
